Roadmap

From a workstation to a supercomputer.

Two roadmaps in one. The first is set by physics and participation: how much compute we unlock as nodes join — and what each scale makes possible. The second is set by us: the platform we are building on top of the mesh — durable storage, replication with no single point, custom-domain TLS, more runtimes, GPU, and an economy that pays the hosts. Numbers below are honest and deliberately approximate.

Track 1 · Compute

What each scale unlocks.

Every device that opens /node adds its cores, memory, and (where present) its GPU to one pool. The aggregate climbs the ladder below. These are order-of-magnitude estimates from typical consumer hardware, not benchmarks — the live figure is on the network page.

1 – 10nodes
A single workstation
Tens of cores, a handful of GPUs. Enough to run the demos live: WASM and JavaScript tasks, a hosted app with a database and realtime rooms, content-addressed blobs. This is where the mesh is today.
✓ we are here
~100nodes
A server rack
Hundreds of cores across cities. Embarrassingly-parallel batch work at useful throughput: sharded CI, render fan-out, parameter sweeps, bulk inference on donated GPUs — the work splits, scatters, and comes back.
~1,000nodes
A small datacenter
Thousands of cores and meaningful GPU memory. Distributed training and large simulation ensembles become practical; the credit economy starts pricing real, scarce capacity instead of demo cycles.
~10,000nodes
Petaflop-class aggregate
At ten thousand mixed devices the pooled floating-point throughput reaches the petaflop range — the territory of a national research cluster, owned by no one and rented from everyone. Population-scale genomics, climate ensembles, full-network LLM inference.
100k – 1Mnodes
Rivals a top-500 supercomputer
At hundreds of thousands to a million phones, laptops, and GPUs, the aggregate enters the regime of the world's largest machines — a planetary instrument for science that anyone can join in one click and anyone can point at the hardest problems.

Approximate by design. Real throughput depends on device mix, GPU share, network latency, and how parallel the workload is — a million phones is not a million server cores. We measure and publish the live aggregate on the network page rather than promise a fixed number here.

Track 2 · Platform

What we are building.

CE is a global compute mesh that is also a one-command app platform: hosting, a database, realtime rooms, custom domains, one-identity signing, an authoritative game framework (netgame + drift), and a public projects registry, hot-reloaded on a public dev domain. Here is where that platform is going — grouped by horizon, concrete but not over-promised.

Short termshipping now
✓
One-command app platform
npm create ce-app, npx ce-app dev → live URL with hosting, KV database, realtime rooms, and hot-reload on save. Live today.
✓
One-identity signing, verified
ce-app resolves one Ed25519 identity (the CE node key, found cross-platform) and signs every mutating write. The hub now verifies it: signed writes are attributed to an owner and earn identity-scoped quotas; unsigned stays anonymous on conservative caps.
✓
Authoritative game framework (netgame + drift)
netgame elects an authoritative host over live mesh metrics with /db-snapshot failover. drift — a wasm + wgpu open world — is live at drift.ce-net.com with region sharding and host failover; coop.ce-net.com is the worked example.
✓
Slugs, registry & feedback
Claim a human slug (/apps/<slug>/), publish to the public projects registry (ce-net.com/projects and /apps render it), and collect feedback at feedback.ce-net.com. Signed, owner-scoped; identity-based admin moderation.
✓
Custom production domains
CNAME a domain to ce-net.com and ce-app domain add binds it to your app. Per-developer subdomains via wildcard DNS are live.
✓
Build on the relay
deploy/ce-build.sh builds ce-hub and Rust→wasm apps (drift) natively on the relay — the same target we deploy to — and PUTs assets straight to /apps/<id>/. No laptop cross-compile.
→
Database durability
Persisted, content-addressed snapshots so the KV store survives restarts and grows past a single host's memory. Compare-and-set (If-Match) is live; durable backing next.
→
Custom-domain TLS automation
Automatic certificate issuance and renewal for CNAME'd domains, so a custom domain is one command end to end.
→
More languages & runtimes
Compute tasks run WASM, JavaScript, and Python today. Broadening the sandboxed runtime set and the task SDK surface.
Medium termin design
→
Mesh replication & read-path
App files, blobs, and database snapshots replicated across contributor nodes, served from the nearest replica — a real distributed read-path, not a single origin.
°
Scheduler-placed game hosting
Today netgame elects the authoritative host from emergent mesh metrics. A future hub scheduler could place hosting on a chosen node by trust, region, or capacity and hand clients the host id directly — the wire protocol already leaves room for it.
°
Multi-relay, no single point
More than one public relay and ingress, so no one machine is the chokepoint. The mesh stays reachable if any single relay goes down.
°
GPU compute via WebGPU
Browser and headless nodes that expose WebGPU run GPU tasks directly — inference and parallel kernels on donated graphics hardware, dispatched like any other task.
°
Capability-based auth across the platform
App data, databases, and rooms gated by signed, attenuating capabilities — delegate, narrow, and revoke access without an account system.
°
Paid hosting via the credit economy
Hosting, storage, and compute settle in credits so the nodes that hold your app and run your tasks are paid for it — the same ledger that mints uptime rewards.
Long termdirection
°
Economic security: bonds & slashing
Hosts post bonds and stake reputation; misbehavior is slashed on-chain. A verification dial lets a payer trade cost for redundancy and proof.
°
Fully decentralized ingress
Any node can serve as ingress for an app or domain, with discovery and TLS handled by the mesh — removing the last hosted dependency.
°
Confidential & verifiable compute
Trusted-execution and proof-carrying tasks so sensitive workloads run on hardware you do not own, with guarantees you can check.
°
A self-sustaining app commons
Developers ship apps in one command; users run nodes in one click; the economy keeps the lights on — a public platform that funds and hosts itself.
