Build on CE

CE is a trustless, Byzantine-fault-tolerant compute marketplace on a proof-of-work chain. Run a node, mine credits, and spend them to run containers on machines anywhere — no accounts, no orchestrator, no middleman. Bitcoin, but it runs Docker.

CE is three layers, and you only touch the ones you need:

  • The node — one binary (ce) that joins the mesh, mines, settles payments, and runs sandboxed jobs. It owns the trustless mechanism: identity, transport, ledger, capabilities.
  • The SDK — typed crates that talk to your local node. ce-rs (core: economy, placement, messaging) and ce-coord (optional: streams + replicated state). Additive — add what you need.
  • Apps — programs built on the SDK that own policy and UX: swarm (scheduler), rdev (remote dev). New device-to-device features are apps, not node changes.
Mental model. The node is the part that trusts no one. Everything that assumes cooperation — schedulers, replicated state, dashboards — lives above it in the SDK/app tiers. That boundary is the whole architecture.

Quickstart: run a node

Start a node. It auto-joins the public mesh via ce-net.com/bootstrap, begins mining, and exposes a local HTTP API on :8844.

# start the node (foreground; auto-bootstraps + mines)
ce start

# in another shell — node id, chain height, balance
ce status
ce id

Behind NAT? ce start reserves a relay circuit automatically — no flags needed. Once a few blocks land, your balance grows from the per-block uptime reward. That balance is what you spend to run jobs on other machines.

Docker is optional. Without it, a node still mesh-routes, mines, and participates in the economy — it just can't host container jobs.

Identity

A node's identity is an Ed25519 keypair. The 32-byte public key is the NodeId — the principal for everything: signing transactions, authoring messages, holding capabilities. There is no registration and no account system; you generate a key and you are a participant.

The secret key lives at ~/.local/share/ce/identity/node.key (chmod 600). Back it up — it is your node, your balance, and your reputation.

The mesh

Nodes talk over a libp2p mesh — never stored IP:port. The stack:

  • Transport: TCP + QUIC, Noise-encrypted (the sender's NodeId is authenticated at the wire).
  • Discovery: Kademlia DHT + mDNS, bootstrapped from ce-net.com/bootstrap.
  • NAT traversal: AutoNAT + circuit relay + DCUtR hole-punching.
  • Pub/sub: gossipsub topics for transactions, blocks, sync, CEP-1 cell signals, and app topics.
  • Directed: a request/reply RPC protocol (/ce/rpc/1) and raw streams (tunnel).

Because the sender of every message is cryptographically authenticated, apps get "who said this" for free — the foundation the SDK builds streams and replicated state on.

Ledger & credits

A proof-of-work chain orders all economic activity. Nodes mine a block roughly every 10 seconds and earn an uptime reward in credits.

  • Emission: starts at 1,000 credits/block, halves every 210,000 blocks, hard cap 21 billion credits.
  • Money is integer base units — never floats. 1 credit = 10^18 base units (wei-style). On-chain amounts are u128; the HTTP API carries them as decimal strings.
  • Jobs: a JobBid escrows credits from the payer; JobSettle moves the cost to the host once work is delivered. No balance → 402.
  • Channels: off-chain payment channels meter long-running and data-layer work with signed receipts.

Every settlement burns 1% — a small, constant Sybil tax that makes wash-trading reputation expensive.

Capabilities

Authorization is not an allowlist — it's a signed, attenuating capability chain (the ce-cap crate). A resource owner issues a capability to another node; that node may delegate a narrower one onward. Each link can only shrink authority.

# on the DESKTOP — issue a capability to the laptop
ce grant <laptop-node-id> --can exec,sync,tunnel --expires 90d
# → prints a capability token the laptop stores and presents

Abilities are opaque strings (exec, sync, tunnel, deploy…). A node honors any chain rooted at its own key or a configured root. Revocation is on-chain (RevokeCapability) plus expiry — no network lookup on the hot path.

Compute

CE runs two kinds of workload, both sandboxed and metered:

  • Docker (via bollard) — pulled images, CPU/memory limits, network_mode: none (all traffic routes through the mesh), and gVisor (runsc) isolation when available.
  • WASM (via wasmtime) — content-addressed modules, fuel-metered, memory-capped, deterministic. Ideal for verifiable redundant execution.

Hosts meter usage and bill in 30-second intervals via heartbeats; you're charged for exactly what you use.

The SDK — ce-rs

A typed, async client for your local node. Point it at 127.0.0.1:8844 and go.

use ce_rs::{CeClient, BidSpec, Amount};

let ce = CeClient::local();
let me = ce.status().await?;
println!("height {} · balance {}", me.height, me.balance);

// find a GPU host and deploy a container on it, paid in credits
let host = ce.atlas().await?.into_iter().find(|h| h.has_tag("gpu"))?;
let spec = BidSpec{ image:"alpine".into(), cmd:vec!["echo".into(),"hi".into()],
                     cpu_cores:1, mem_mb:128, duration_secs:60, bid:Amount::from_credits(10) };
let job = ce.mesh_deploy(&host.node_id, &spec, None).await?;

What ce-rs covers: status · atlas · transfer · bid/mesh_deploy · jobs · blobs & objects · payment channels · publish/subscribe/request/reply · naming & discovery. Amounts use an Amount type that serializes as a decimal string (JSON can't hold 10^18-scale integers).

Add it: ce-rs = { git = "https://github.com/ce-net/ce-rs" }

Coordination — ce-coord

The coordination layer CE deliberately keeps out of the node: typed pub/sub streams and replicated collections, built as a library over ce-rs. It's fully decoupled — same client, additive crate.

use ce_coord::Coord;

let coord = Coord::with_client(ce.clone()).await?;  // shares the ce-rs client

// typed pub/sub: publish on one node, receive on others
let mut s = coord.stream::<Reading>("sensors").await?;
s.publish(&reading).await?;

// replicated map: one writer, many readers, versioned convergence
let map = coord.map_writer::<String,i64>("balances").await?;
let v = map.insert("alice".into(), 100).await?;

// on a reader node (knows the writer's NodeId):
let map = coord.map_reader::<String,i64>("balances", writer_id).await?;
map.await_version(v).await;            // converged
assert_eq!(map.get(&"alice".into()), Some(100));

PrimitiveWhat it is

How it works & scales

A writer broadcasts versioned operations on a gossip topic; readers apply them in order and repair any gaps with a directed catch-up request to the writer. Integrity is free — a reader only applies entries whose authenticated sender is the writer it follows.

  • Reads scale flat — every reader answers from its own local replica, zero writer load.
  • A write is one broadcast — cost independent of reader count.
  • Catch-up is O(missed), not O(log) — steady-state readers never ask.
v0 is single-writer (a fixed leader). Multi-writer Raft leader election, blob-store snapshots, SSE push, and ce-cap-gated membership are the roadmap — and the Replicated engine already isolates the writer role, so collection call sites won't change.
Add it: ce-coord = { git = "https://github.com/ce-net/ce-coord" } — depends on ce-rs, never the reverse.

Apps: swarm & rdev

Apps are ordinary programs on the SDK — they own policy, CE owns mechanism.

# swarm — fan a command across hosts, gather results
swarm run alpine -n 8 --select gpu -- nvidia-smi
swarm verify alpine -k 3 -- sha256sum /etc/os-release   # redundant, compared

# rdev — remote dev over the mesh, gated by capabilities
rdev exec desktop --image alpine -- uname -a
rdev watch ~/ce-net desktop:ce-net                       # continuous mirror

FAQ

Is CE Byzantine-fault-tolerant?

Yes — that's the point. The node assumes adversaries and uses proof-of-work, economic escrow, settlement burns, and on-chain reputation for Sybil resistance. (This is the key difference from honest-fleet runtimes like Flashbots' Mosaik, which assume cooperating members and aren't BFT.)

How is this different from Kubernetes or Ray?

Those orchestrate machines you own. CE is a marketplace between parties who don't trust each other, settled in credits, with no central control plane. swarm gives you Ray-like scatter/gather on top.

Do I need credits to start?

No — run a node and mine them. You only need a balance to spend on others' compute.

Where do new device-to-device features go?

In apps/SDK crates over the mesh primitives (pub/sub + request/reply + ce-cap), not as new node RPCs. The node stays small and trustless.
