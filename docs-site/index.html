CE Platform · live at ce-net.com

Build and ship on CE.

CE is a mesh of compute nodes — phones, laptops, and servers running in the browser or headless. On top of that mesh sits a developer platform: deploy any frontend with one command, store data in a persistent database, broadcast realtime messages, and run compute tasks in WebAssembly, JavaScript, or Python. Everything in these docs is deployed and reachable today at https://ce-net.com.

Overview

CE has two halves that share one network:

  • The mesh — compute nodes that anyone can run. Open ce-net.com/node in any browser (including a phone) and the tab becomes a real WebAssembly/JavaScript compute node and mesh peer. A headless node is one command on a server. Nodes advertise their cores, RAM, storage, and GPU; they receive work and return results.
  • The hub — the developer-facing service (ce-hub) exposed publicly through nginx at ce-net.com. It is one HTTP surface that gives you: app hosting, a persistent database, realtime rooms, content-addressed blob storage, and a compute task router that dispatches your tasks to connected nodes.

You build against the hub. The hub talks to the mesh. Every API in this document is served from https://ce-net.com and is also reachable under the /hub/<path> prefix.

One origin, three ways to reach it. Your app, its database, and its rooms are all available on the path origin https://ce-net.com/apps/<id>/, on your personal subdomain https://<id>.ce-net.com/, and on any custom domain you CNAME to ce-net.com.

Quickstart

The fastest path is the ce-app CLI. It scaffolds a template, detects your framework, builds it, uploads the result to the hub, and prints your live URL — then hot-reloads on every save.

One-command developer flow

# scaffold the chat template into a new directory
npm create ce-app

# or pick a template explicitly
ce-app new vite-react my-app

# develop: auto-detect framework, build, upload, print URL, hot-reload on save
cd my-app && npx ce-app dev

# one-shot deploy without the watcher
npx ce-app deploy

# attach a custom production domain
npx ce-app domain add app.acme.com

ce-app dev prints a URL like https://ce-net.com/apps/<your-id>/. Your app id (your public developer id) is generated once and stored at ./.ce/app-id, so every later command targets the same app.

Available templates: chat, notes, board, vite-react, svelte, react, next, react-native.

Deploy without the CLI

The CLI is just a wrapper over the HTTP API. You can upload a built site with PUT requests and it goes live immediately. The empty path is your index.html.

# upload index.html — empty path means the app root
curl -X PUT https://ce-net.com/apps/my-app/ \
  -H "content-type: text/html" \
  --data-binary @dist/index.html

# upload an asset
curl -X PUT https://ce-net.com/apps/my-app/assets/app.js \
  -H "content-type: application/javascript" \
  --data-binary @dist/assets/app.js

# live now
open https://ce-net.com/apps/my-app/
const base = "https://ce-net.com";
const id = "my-app";

await fetch(`${base}/apps/${id}/`, {
  method: "PUT",
  headers: { "content-type": "text/html" },
  body: "<!doctype html><h1>hello from CE</h1>",
});

// -> { id: "my-app", path: "index.html", version: 1, url: "/apps/my-app/" }

Hosting & frontend frameworks

CE hosts static sites and single-page apps. You upload built files; the hub serves them with the right content type, applies SPA fallback, and injects a hot-reload snippet into HTML. There is no build step on the server — you build locally (or the CLI does) and upload the output.

SPA fallback

Client-side routers need any unknown route to return index.html. Turn that on per app:

curl -X PUT https://ce-net.com/apps/my-app/config \
  -H "content-type: application/json" \
  -d '{"spa":true}'

# inspect
curl https://ce-net.com/apps/my-app/config
# -> {"spa":true,"version":4,"domains":["app.acme.com"]}
await fetch("https://ce-net.com/apps/my-app/config", {
  method: "PUT",
  headers: { "content-type": "application/json" },
  body: JSON.stringify({ spa: true }),
});

With SPA enabled, a route-like miss that has no file extension returns index.html with a 200 status, so deep links and refreshes work. Requests for real assets (paths with an extension) still 404 if missing, so a broken script tag is obvious.

Content types & the root file

Each PUT /apps/:id/*path stores one file at that path with whatever content-type you send. An empty path stores index.html. The response includes the incremented version and the public url.

CE database

A persistent key-value store, namespaced per app. Values are arbitrary JSON. Lists come back newest-first and can be filtered by prefix. No schema, no setup — the namespace is your app id.

Set, get, delete a key

# write JSON to a key
curl -X PUT https://ce-net.com/db/my-app/user:42 \
  -H "content-type: application/json" \
  -d '{"name":"Ada","score":9}'
# -> {"ok":true,"key":"user:42"}

# read it back
curl https://ce-net.com/db/my-app/user:42
# -> {"name":"Ada","score":9}   (or 404 if absent)

# list newest-first, filter by prefix, cap the count
curl "https://ce-net.com/db/my-app?prefix=user:&limit=20"
# -> {"items":[{"key":"user:42","value":{...}}],"n":1}

# delete
curl -X DELETE https://ce-net.com/db/my-app/user:42
import { createClient } from "@ce/client";
const ce = createClient();   // appId auto-resolves from the page URL

await ce.db.set("user:42", { name: "Ada", score: 9 });
const u   = await ce.db.get("user:42");
const all = await ce.db.list("user:", 20);
await ce.db.del("user:42");
const base = "https://ce-net.com", app = "my-app";

await fetch(`${base}/db/${app}/user:42`, {
  method: "PUT",
  headers: { "content-type": "application/json" },
  body: JSON.stringify({ name: "Ada", score: 9 }),
});

const u = await (await fetch(`${base}/db/${app}/user:42`)).json();
Listing semantics. GET /db/:app returns {items:[{key,value}],n}, newest-first. prefix filters keys; limit caps how many come back. Useful for feeds, message logs, and leaderboards where the most recent writes matter most.

The database is reachable from all three origins for your app — the path origin, your subdomain, and any custom domain — so a page served at app.acme.com can call /db/my-app/<key> on the same host.

Realtime rooms

Rooms are WebSocket channels under an app. Connect to /rt/:app/:room; any text frame you send is broadcast to every other connection in that room. On connect, the last 50 messages are replayed to you, so a client that joins late still sees recent history.

import { createClient } from "@ce/client";
const ce = createClient();

const room = ce.room("lobby");
room.onOpen(() => room.send(JSON.stringify({ join: "Ada" })));
room.on((msg) => console.log("got", msg));   // fires for the 50-message replay too
// room.send(text) -> broadcast to others; room.close() to leave
// works on ce-net.com, <id>.ce-net.com, and your custom domain
const ws = new WebSocket("wss://ce-net.com/rt/my-app/lobby");

ws.onopen    = () => ws.send("hello room");
ws.onmessage = (e) => console.log(e.data);   // last 50 replayed first, then live

Messages are opaque text — send plain strings or JSON you stringify yourself. Pair rooms with the database when you want presence-style ephemerality (rooms) plus durable history (database).

Compute tasks

POST /tasks hands a unit of work to the mesh. The hub picks a connected node, runs it, and returns the result with timing and which node ran it. Two shapes are accepted: a named WebAssembly function, or inline code in a sandboxed language.

WASM task

curl -X POST https://ce-net.com/tasks \
  -H "content-type: application/json" \
  -d '{"func":"count_primes","args":[200000],"ret":"i32","module":"demo"}'
# -> {"node":"...","lang":"wasm","func":"count_primes","ok":true,"value":17984,"ms":42}
const r = await fetch("https://ce-net.com/tasks", {
  method: "POST",
  headers: { "content-type": "application/json" },
  body: JSON.stringify({
    func: "count_primes", args: [200000], ret: "i32", module: "demo",
  }),
});
console.log(await r.json());

Code task (js or python)

curl -X POST https://ce-net.com/tasks \
  -H "content-type: application/json" \
  -d '{"lang":"js","code":"i=>i.a+i.b","input":{"a":2,"b":3},"func":"task"}'
# -> {"node":"...","lang":"js","func":"task","ok":true,"value":5,"ms":3}
const r = await fetch("https://ce-net.com/tasks", {
  method: "POST",
  headers: { "content-type": "application/json" },
  body: JSON.stringify({
    lang: "js",
    code: "i => i.a + i.b",   // runs in a sandboxed Worker on a node
    input: { a: 2, b: 3 },
    func: "task",
  }),
});
console.log(await r.json());   // { ok:true, value:5, ms:3, node:"..." }
# python runs under Pyodide — browser nodes only
curl -X POST https://ce-net.com/tasks \
  -H "content-type: application/json" \
  -d '{"lang":"python","code":"def task(i):\n    return i[\"a\"] + i[\"b\"]","input":{"a":2,"b":3},"func":"task"}'

Targeting a specific node

Add an optional "target":"<full node id>" to route a task to one node (find ids via GET /nodes). Without a target the hub chooses.

Status codes. POST /tasks returns 503 if no nodes are connected, and 504 if the chosen node did not return a result in time. The success body is always {node, lang, func, ok, value, ms, error} — check ok and read error when it is false.

Custom domains

Point a production domain at your app in three steps: attach the domain, create a CNAME, and you are live with TLS.

  1. Attach the domain to your app (CLI or API). The response tells you the CNAME target — always ce-net.com.
  2. At your DNS provider, create a CNAME for the domain pointing to ce-net.com.
  3. Load https://<your-domain>/. nginx sets an X-CE-Host header so the hub resolves which app to serve via GET /_host/*path.
npx ce-app domain add app.acme.com
# prints the CNAME (app.acme.com -> ce-net.com) and the TLS note
# attach
curl -X PUT https://ce-net.com/apps/my-app/domain \
  -H "content-type: application/json" \
  -d '{"domain":"app.acme.com"}'
# -> {"domain":"app.acme.com","id":"my-app","cname":"ce-net.com"}

# list all mapped domains
curl https://ce-net.com/domains
# -> [{"domain":"app.acme.com","id":"my-app"}]

# detach
curl -X DELETE https://ce-net.com/apps/my-app/domain/app.acme.com

Your database and rooms work on the custom origin too: https://app.acme.com/db/my-app/<key> and wss://app.acme.com/rt/my-app/<room>.

Your subdomain

Every app id also resolves on a wildcard subdomain, live today: https://<id>.ce-net.com/. No setup — deploy to an id and the subdomain serves it. The @ce/client SDK auto-resolves your app id from either the /apps/<id>/ path or the <id>.ce-net.com host, so the same client code works on all three origins.

Hot reload

Every HTML page the hub serves gets a small hot-reload snippet injected. The snippet subscribes to GET /apps/:id/__reload, a Server-Sent Events stream that emits the app's version number on every change. When you upload a new file (the version increments), the page reloads automatically.

ce-app dev wires this for you: save a file, the CLI rebuilds and re-uploads, the version bumps, and the open tab refreshes. To watch reloads yourself:

curl -N https://ce-net.com/apps/my-app/__reload
# SSE stream; emits the new version number on each upload

Uptime & latency metrics

The mesh reports live aggregate metrics. GET /stats is a one-shot snapshot; GET /stats/stream pushes the same object as SSE roughly every 1.5 seconds. The human view is at ce-net.com/network.

Contributor caches

Contributor nodes cap the replicated-file cache they hold, oldest-evicted: a browser node defaults to 32 MiB (tunable via ?cache=), a headless worker to 64 MiB (via CE_WORKER_MAX_CACHE_BYTES). Full detail lives in web/deploy/storage-limits.md.

@ce/client

The browser ESM client (web/ce-app/client) is the recommended way to talk to the hub from a page. createClient() resolves your appId automatically from the /apps/<id>/ path or the <id>.ce-net.com host.

import { createClient } from "@ce/client";

const ce = createClient();
ce.appId;   // resolved app id
ce.base;    // API base origin

// database
await ce.db.set(key, value);
await ce.db.get(key);
await ce.db.del(key);
await ce.db.list(prefix, limit);

// realtime
const room = ce.room(name);
room.send(text);
room.on((msg) => { });
room.onOpen(() => { });
room.close();

Run a node

Contribute capacity to the mesh in two ways:

Browser node

Open https://ce-net.com/node — the tab becomes a real WASM/JS compute node and mesh peer. Works on iOS and Android too.

Headless node

curl -fsSL https://ce-net.com/worker.js | node - --hub wss://ce-net.com/hub

The headless worker runs wasm and js tasks. Nodes connect over the GET /node WebSocket: they send hello, hb (heartbeat), pong, and result frames upstream; the hub sends job, ping, store, and fetch frames downstream.

HTTP API reference

Base origin https://ce-net.com. Every endpoint is also reachable under /hub/<path>. Bodies are JSON unless noted; blob and file uploads take a raw body with a content-type.

Compute

POST/tasks

Run a compute task on the mesh. WASM shape: {"func":"count_primes","args":[200000],"ret":"i32","module":"demo"}. Code shape: {"lang":"js"|"python","code":"i=>i.a+i.b","input":{...},"func":"task"}. Optional "target":"<full node id>" pins a node.

Returns {node, lang, func, ok, value, ms, error}. 503 if no nodes are connected; 504 if the chosen node did not return in time.

GET/stats

Aggregate mesh metrics: nodes, cores, ram_gb, storage_gb, gpus[], gpu_count, gpu_vram_gb, webgpu_nodes, perf_score, tasks_run, blobs, blob_bytes, avg_uptime_pct, by_class{phone,laptop,server,device}, most_reliable{short,uptime_pct}, latency_p50_ms, latency_p95_ms, latency_measured, data_used_bytes, hosted_apps, replicas, limits{max_data_bytes,max_apps,max_domains,max_db_namespaces}.

GET/stats/stream

Server-Sent Events of the /stats object, roughly every 1.5 seconds.

GET/nodes

Connected nodes: {nodes:[{id, short, cores, ram_gb, storage_gb, gpu, webgpu, platform, tasks_run, age_s, rtt_ms, uptime_pct, device_class, online_secs, sessions, first_seen_unix}]}.

GET/node

WebSocket endpoint for browser and headless nodes. Upstream frames: hello, hb, pong, result. Downstream frames: job, ping, store, fetch.

Content-addressed blobs

PUT/blobs/:hash

Store a blob. Raw body plus a content-type header. Returns {hash, size}.

curl -X PUT https://ce-net.com/blobs/<hash> \
  -H "content-type: application/octet-stream" \
  --data-binary @file.bin
# -> {"hash":"...","size":1234}
GET/blobs/:hash

Fetch a blob by its content hash.

App hosting (static sites + SPAs)

PUT/apps/:id/*path

Upload a file. Raw body plus content-type. Empty path stores index.html. Returns {id, path, version, url}.

GET/apps/:id  /apps/:id/  /apps/:id/*path

Serve app files. SPA fallback: a route-like miss with no file extension returns index.html with a 200. HTML responses get the hot-reload snippet injected.

GET/apps/:id/__reload

Server-Sent Events; emits the app's version number on every change.

PUT/apps/:id/config  GET/apps/:id/config

PUT with {"spa":true} toggles SPA fallback. GET returns {spa, version, domains}.

Custom production domains

PUT/apps/:id/domain

Attach a domain: {"domain":"app.acme.com"}. Returns {domain, id, cname:"ce-net.com"}. Create the CNAME at your DNS provider.

DELETE/apps/:id/domain/:domain

Detach a domain from an app.

GET/domains

List all domain mappings: [{domain, id}].

GET/_host/*path

Resolves the app from the X-CE-Host header, which nginx sets for CNAME'd custom domains. You do not call this directly; it powers custom-domain serving.

CE database (persistent KV)

PUT/db/:app/:key

Store a JSON value at a key. Returns {ok, key}.

GET/db/:app/:key

Read a value, or 404 if the key is absent.

DELETE/db/:app/:key

Delete a key.

GET/db/:app?prefix=&limit=

List keys newest-first: {items:[{key, value}], n}. prefix filters keys; limit caps the count.

Realtime rooms

GET/rt/:app/:room

WebSocket. Any text frame is broadcast to others in the room. The last 50 messages are replayed on connect. Reachable as wss://ce-net.com/rt/:app/:room (and on the subdomain / custom domain origins).
